## Методика v3 (июль 2026) — план на месяц через столбец O

**Источник истины:** столбец O (Накопительный объем за месяц → План) в шаблоне ЕЖО.

### План на месяц (начало месяца)
1. `Алихан раскрой отчет` — бот присылает шаблон со ВСЕМИ раскрытыми строками
2. Пользователь заполняет столбец **O** (колонка 15) по кодам где идут работы
3. Отправляет заполненный шаблон в песочницу
4. Бот читает O: строка в опросе если O > 0

### Фильтр опроса (poll.py — 16.07.2026)
- **Единственное условие:** столбец O (колонка 15) > 0
- Остаток (U, колонка 21) — только для отображения, не фильтр
- Здания выводятся динамически — любые названия из шаблона (Благоустройство, НК, НТ и др.)
- Сообщение разбито на 2 части (header + residuals) под лимит WhatsApp
- Никаких schedule-based догадок и запасных формул расчёта остатка

### Pipeline v3

1. **MONTHLY** — в начале месяца «Алихан раскрой отчет» → заполнить O
2. **POLL** — «Алихан запускай опрос» → бот читает O, показывает ✅/❌ по QA + остатки
3. **FOREMAN** — прорабы отвечают: `код = объём`
4. **CLOSE** — «Алихан закончить опрос» → автозаполнение → fill_ejo → отправка
5. **VERIFY** — проверить отчёт через openpyxl. Не прошёл — не отправлять

5. `systemctl --user restart alikhan.service`
6. Проверить: `tail -5 /tmp/alikhan.log` — бот работает с реальной датой
7. Шаблон с фото вырастает с ~300KB до ~1.3MB — это нормально

## QA-парсер (qa.py)

Формат данных от прораба:
- «Атантай ИТР 1, рабочих 6» → 2 факта: «Атантай ИТР 1», «Атантай 6 рабочих»
- «2.2.3.2 = 77.52» → бетонирование факт
- VOR-коды: поддерживает `=`, `—`, `–`, `-` как разделители
- is_qa: триггеры + VOR-коды + contractor keywords
  - Полный список триггеров: айбикон, атантай, майкадам, наватек, итр, рабочих, водител, происшестви, сделано, не успели, техник, материал, постав, документац, план на, план работ
- **«нет»-паттерны:** Grok не парсит короткие «Материалов нет»/«Техники нет» → 0 фактов. `_parse_no_patterns()` regex-фолбэк ловит их ДО Grok (см. qa.py строки 24-37)

## fill_ejo.py — заполнение (v3, 2026-07-10)

- **Прямой запуск:** `python3 fill_ejo.py YYYY-MM-DD` (без EJO_DATE, CLI-аргумент)
- **Зависимости:** `pip install openpyxl Pillow` (могут отсутствовать в свежем venv)
- **Формат:** 3 листа (без отдельного «Фототчет») — фото в строках 856-859 Листа 1
- **Жёлтая заливка:** определяется из `bot_schedule_phases` (активные разделы из БД) + объёмы за сегодня. FFFF00 на всю строку A-U.
- **Готовность K853:** `calc_completion_pct()` — sum(S)/sum(K)×100 по всем строкам с кодом ВОР
- staff(date): читает qa(date, 'персонал'), парсит contractor + ИТР/рабочие. Поддерживает раздельные факты и совмещённые
- volumes(date): qa(date, 'бетонирование') + qa(date, 'монтаж') + qa(date, 'земляные работы'). VOR-коды с любым разделителем
- photos(date): из bot_memory_messages по DATE(created_at). Вставляются в строки 856-859 Листа 1
- plans: из QA-фактов с ключевым словом «план»

### Симуляция без QA-фактов — ожидаемое поведение
При запуске за дату без QA-фактов (например симуляция будущего дня):
- Персонал = 0 по всем подрядчикам (Атантай, Майкадам, Наватек)
- Инциденты = 0
- Объёмы за сутки = пустые
- Погода = заполняется (Open-Meteo всегда доступен)
Это корректно — скрипт не падает, возвращает чистый заполненный файл.

## Исправленный ЕЖО → новый шаблон (2026-06-29)

При получении документа с «ЕЖО» в имени:
1. `_update_template_from_correction()` в `main_waha.py` сравнивает с авто-сгенерированным **по ВОР-коду (col C), а не по номеру строки** — иначе вставка/удаление строк ломает diff (672 ложных отличия вместо реальных)
2. **Дату извлекать из имени файла:** `re.search(r'(\d{2})\.(\d{2})\.(\d{4})', fname)` → `YYYY-MM-DD`. Не `datetime.now()` — файл за 27.06, а сегодня 29.06.
3. Выводит отличия в группу
4. Заменяет `templates/ЕЖО_шаблон.xlsx` на исправленный
5. Сохраняет копию как `/tmp/ЕЖО_{date}_v1.xlsx` — для цепочки `yesterday_cum()` на следующий день
6. `fill_ejo.py`: при v>0 (есть сегодняшние работы) берёт `prev_p` из `yesterday_cum()`, а не из шаблона — иначе удвоение (шаблон уже содержит сегодняшний итог после правки)
7. Для кодов БЕЗ сегодняшних работ: `prev_p = max(шаблон, yesterday_cum)`
8. **Bootstrapping:** первый прогон требует исправленный файл за предыдущий день. Если нет — yesterday_cum вернёт авто-значения. Пользователь должен прислать исправленный файл за вчера.

Логотип IBCON встроен в шаблон — не добавлять программно в `fill_ejo.py`.

## Pitfalls

### SIM_DATE vs datetime.now()
Все компоненты (main_waha, router, smart_evening_check, fill_ejo) используют SIM_DATE/EJO_DATE. Если хотя бы один использует datetime.now() — данные уйдут на неверную дату. Проверять grep datetime.now() перед тестом.

### QA-парсер теряет contractor names
Grok-парсер в qa.py может разделить «Атантай ИТР 1, рабочих 6» на «ИТР 1» и «рабочих 6» без contractor. Промпт должен явно требовать contractor в каждом факте.

### fill_ejo падает на «без тег»
Фото без caption сохраняются с building="без тег". fill_ejo должен: `if bld in ('без тег', 'без тега'): bld = 'Общий план'`

### VOR-коды с тире вместо =
QA-парсер выдаёт факты с `—` (em dash) вместо `=`. volumes() и планы должны матчить оба разделителя.

### Дубликаты фактов (2026-07-01)
Бот может обработать одно сообщение дважды (seen_id race condition). Дубликаты в `bot_memory_facts` → `staff()` удваивает персонал (Атантай 8→16, Майкадам 28→56). **Фикс:** `ROW_NUMBER() OVER (PARTITION BY fact_date, category, fact)` дедупликация перед fill_ejo. Дубликаты фото — чистить через SQL DISTINCT или ROW_NUMBER OVER PARTITION.

### QA-экстрактор: +1 к рабочим (2026-07-01)
Из «Атантай ИТР 1, рабочих 6» QA выдаёт «Атантай 7 рабочих» (+1 от ИТР). Проверять факты после QA.

### smart_evening_check: план ≠ остаток (2026-07-01)
Функция `_safe_float(col21, col12)` использовала план (col12) как fallback для остатка (col21). Это приводило к тому, что завершённые работы (остаток=None) отображались в опроснике с ложным остатком из плана. **Фикс:** убран fallback — `_safe_float(col21)` без col12. Если остаток None → работа исключается из опросника.

### VERIFY REJECT — агент не сообщает о пустой БД (2026-07-01)
Если данных нет, агент должен СНАЧАЛА сказать «данных нет», ПОТОМ предложить опрос. Иначе Grok-верификатор даёт REJECT <40.

**Фикс в router.py:** добавить CMD action для команд («запускай опрос», «формируй ЕЖО» и др.) ДО Grok/verify:
```python
# 2.5 Command detection — skip Grok/verify for known commands
cmd_words = ["запускай опрос", "начать опрос", "заполни ежо", "сформируй ежо",
             "формируй ежо", "сделай ежо", "закрыть опрос", "завершить опрос",
             "закончить опрос", "стоп опрос"]
if any(w in text.lower() for w in cmd_words):
    return "CMD", "", False
```
В main_waha.py: `if action in ("IGNORE", "CMD"): continue` — команды не идут через Grok.

### psycopg2 % в LIKE с кириллицей
`cur.execute("... ILIKE '%план%'", (today,))` → `IndexError: tuple index out of range`. `%` внутри строки интерпретируется как placeholder. Фикс: `ILIKE '%%план%%'`. Актуально для всех русскоязычных LIKE-запросов в smart_evening_check.py и fill_ejo.py.

### PostgreSQL regex: `[0-9]` не `\d`
`~ E'\\d+\\.\\d+\\.\\d+'` возвращает 0. PostgreSQL POSIX regex не поддерживает `\\d`. Использовать: `~ E'[0-9]+[.][0-9]+[.][0-9]+'`. Подробно в `alikhan-ejo-report/references/postgresql-regex-patterns.md`.

### Факты сохраняются с неверной датой (2026-07-01)
Даже при `SIM_DATE = None` бот может сохранить QA-факты с датой вчерашнего дня. Причина: `datetime.now()` в UTC может отставать от местного времени (Бишкек UTC+6). Проверять ДО fill_ejo:
```bash
python3 -c "
from db import get_conn; import psycopg2.extras
c = get_conn().cursor(cursor_factory=psycopg2.extras.RealDictCursor)
c.execute(\"SELECT fact_date, category, fact FROM bot_memory_facts WHERE source='qa' AND created_at > now() - interval '6 hours' ORDER BY created_at\")
for f in c.fetchall(): print(f\"  [{f['fact_date']}] [{f['category']}] {f['fact']}\")
"
```
Если дата неверная — исправить:
```sql
UPDATE bot_memory_facts SET fact_date='2026-07-01'
WHERE fact_date='2026-06-30' AND created_at > '2026-07-01 18:00:00+00';
```

В fill_ejo.py после записи значений работы: `PatternFill(start_color=Color(rgb='FFFF00'), ...)` на всю строку. Перед этим — полная очистка всех жёлтых в строках 24+ (final sweep).

### fill_ejo.py: планы во ВСЕХ категориях (2026-07-01)
Строка `pf = [x['fact'] for x in qa(date, 'бетонирование') ...]` — ищет планы только в бетонировании. Те же планы в `монтаж` не видны. Фикс: `qa(date)` без аргумента category.

### Разделение планов по зданиям (2026-07-01)
Планы в листе «Материалы и планы» должны иметь заголовки зданий: «1 АБК», «2 Общежитие», «3 Галерея». `bld_rows` переписан с dict на list of tuples с header row (start_row - 1).

### Задвоение дат в материалах (2026-07-01)
`re.sub` заменяет дату, но при `old_v == new_label` (дата уже правильная) код добавлял вторую дату. Фикс: проверка `if not _re.search(date_pattern, new_label)` вместо `if old_v == new_label`.

### Ветер: запятая вместо точки (2026-07-01)
Русский формат: `str(ws).replace('.', ',')` в weather(). Иначе «12.2 км/ч» вместо «12,2 км/ч».

### Погода: force-write (2026-07-01)

### Не менять логику опроса без явного ТЗ (16.07.2026)
