# QA Parser Fixes — 2026-06-29

## Bug: «Материалов нет» silently dropped

### Symptom
`[QA] 0 facts from 'Материалов нет'` — сообщение принято, но ноль фактов сохранено.

### Root Cause (two-layer)

1. **is_qa() trigger gap:** триггеры не включали «материал». Сообщение не попадало в QA → уходило в Grok-роутер как обычный текст.
2. **Grok blind spot:** даже после попадания в QA, Grok возвращал пустой результат для лаконичных «нет»-паттернов.

### Fix

**Layer 1 — is_qa() triggers:**
```python
# qa.py line 15-17
triggers = [..., "материал", "постав", "документац", "план на", "план работ"]
```

**Layer 2 — _parse_no_patterns() fallback:**
```python
# qa.py lines 24-37, invoked at line 44
def _parse_no_patterns(text):
    patterns = [
        (r'материалов?\s*нет', 'общая|документация|Поставок материалов нет'),
        (r'поставок?\s*нет', 'общая|документация|Поставок материалов нет'),
        (r'техник[аи]?\s*нет', 'общая|техника|Техники нет'),
        (r'происшествий?\s*нет', 'общая|инцидент|Происшествий нет'),
        (r'инцидентов?\s*нет', 'общая|инцидент|Происшествий нет'),
    ]
    ...

**Layer 2 v2 — count == 0 fallback (исправлено 29.06):**
Первая версия проверяла `if not result.strip()` — но Grok может вернуть непустой, но бесполезный ответ (комментарий, «нет данных»). Тогда regex-фолбэк не вызывается. **Исправлено:** проверять `count == 0` после парсинга всех строк Grok, а не пустоту результата:
```python
# qa.py: после цикла for line in result.split("\n"):
if count == 0:
    for line in _parse_no_patterns(text).split("\n"):
        parts = [p.strip() for p in line.strip().split("|", 2)]
        if len(parts) >= 3:
            cur.execute("INSERT INTO ...", (gid, today, parts[0], parts[1], parts[2]))
            count += 1
```

### Verification
v1: «Материалов нет» → 1 fact ✅ | «Техники нет» → 0 facts ❌ (Grok вернул непустой ответ)
v2: «Техники нет» → 1 fact ✅

## Bug: smart_evening_check.py IndexError on %план%

### Symptom
```
IndexError: tuple index out of range
```
In `get_qa_status()` line 70: `ILIKE '%план%'` — psycopg2 interprets `%` as placeholder.

### Fix
```python
# WRONG: cur.execute("... fact ILIKE '%план%'", (today,))
# RIGHT:
cur.execute("... fact ILIKE '%%план%%'", (today,))
```

## Bug: VOR-коды классифицируются как «документация»

### Symptom
«Работы 2.4.2.1 = 1870.9» сохраняется с category='документация'. `fill_ejo.py::volumes()` ищет только бетонирование/монтаж/земляные → объём не попадает в суточные.

### Root Cause
Grok видит «Работы» и классифицирует как документацию. VOR-парсер в `parse_qa()` работает на выводе Grok (формат `building|category|fact`), а не на сыром тексте.

### Fix — пре-парсер ДО Grok
```python
# qa.py parse_qa(), перед ask_grok():
for vor_match in re.finditer(r'(\d+\.\d+\.\d+(?:\.\d+)?)\s*=\s*(\d+(?:[.,]\d+)?)', text):
    code, vol = vor_match.groups()
    vol_clean = vol.replace(',', '.')
    cat = 'монтаж' if any(w in text.lower() for w in ['арматур','опал','монтаж','профлист','каркас']) else 'бетонирование'
    cur.execute("INSERT INTO bot_memory_facts (...) VALUES (...,'qa')", (...))
    count += 1
```
Категория определяется по ключевым словам: арматур/опалубка/монтаж/профлист/каркас → монтаж, иначе бетонирование.

## Bug: volumes() и опрос не видят категорию «монтаж»

### Symptom
`fill_ejo.py` не применяет объёмы из QA-фактов категории «монтаж». Опрос показывает «❌ Работы — нет данных» даже при наличии монтажных фактов.

### Fix
- `fill_ejo.py::volumes()`: `qa(date, 'бетонирование') + qa(date, 'монтаж') + qa(date, 'земляные работы')`
- `smart_evening_check.py::get_qa_status()`: `category IN ('бетонирование','монтаж','земляные работы')` → `status['работы']`
- Все ссылки `qa_status['бетонирование']` → `qa_status['работы']`
