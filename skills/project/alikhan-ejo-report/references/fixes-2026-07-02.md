# QA Double-Insertion Bug (Fixed 2026-07-02)

## Symptom
`bot_memory_facts` contained duplicate entries for the same VOR code — one raw (`2.4.2.2 = 2115`) and one labeled (`Выполнено 2.4.2.2 = 2115 кг`). 8 entries instead of 4 for бетонирование category.

## Root Cause
`qa.py` `parse_qa()` had TWO parsing passes on the same text:

1. **Pre-parse (lines 65-73):** `re.finditer` extracted VOR codes directly from raw text and INSERTed as `f'{code} = {vol_clean}'` with source='qa'
2. **Grok parse (lines 74-88):** Grok parsed the same text and returned labeled facts like `Выполнено 2.4.2.2 = 2115 кг`

Both results ended up in the DB.

## Fix
Removed the pre-parse entirely. Grok handles all VOR code parsing with proper labels (`Выполнено`, `План на завтра`). The pre-parse was an optimization that created duplicates.

## Cleanup
```sql
-- Find dupes
SELECT fact, COUNT(*) FROM bot_memory_facts 
WHERE fact_date='2026-07-02' AND category='бетонирование'
GROUP BY fact HAVING COUNT(*)>1;

-- Remove raw entries (keep labeled)
DELETE FROM bot_memory_facts WHERE id IN (4967, 4968, 4969, 4970);
```

## Prevention
After fix, only Grok inserts facts. The `is_qa()` function still catches VOR code patterns to route to QA, but only Grok does the actual extraction and insertion.

# Template Update Cycle (Clarified 2026-07-02)

## Cycle
1. `fill_ejo.py` reads `templates/ЕЖО_шаблон.xlsx` → saves to `/tmp/ЕЖО_{date}_v1.xlsx`
2. User reviews → makes corrections → sends corrected file to WhatsApp
3. `_update_template_from_correction()` compares auto vs corrected → updates template
4. **Template updates ONLY on manual correction.** Auto-generated v1 does NOT replace template.
5. Next day's `fill_ejo.py` reads updated template → loop continues

## When NO corrections
If user says "замечаний нет" — template stays as-is. Auto-generated v1 is used via `yesterday_cum()` for cumulative volumes. Template from previous day remains valid.

# Building Detection Limitation

All QA facts default to `building='общая'`. The parser does not reliably map work codes to buildings. For example, `2.4.2.2` belongs to Общежитие but the parser doesn't know this.

Non-critical for EJO filling: `fill_ejo.py` uses its own code→building map from the template (column A of Sheet 1), not the building field from bot_memory_facts.
