## 1. Diff blind spots in `_update_template_from_correction()`

**File:** `main_waha.py:86-101`

The diff between user-corrected and auto-generated ЕЖО only compares:
- col 16: мес.факт (month cumulative)
- col 19: общ.факт (total cumulative)

## 3. Photo deduplication after seen_ids clear

**Root cause:** Clearing `seen_ids.json` causes the bot to re-process old messages. Each re-processed photo message inserts a duplicate row into `bot_memory_messages`.

**smart_evening_check.py line 74:**
```python
cur.execute("SELECT tags->>'building' as bld, count(*) as cnt FROM bot_memory_messages WHERE message_type='image' AND DATE(created_at)=%s GROUP BY tags->>'building'", (today,))
```
This query counts ALL rows — duplicates inflate the count.

**Real case:** 5 photos of Общежитие → DB had 10 photos (2 АБК, 1 Галерея, 7 Общежитие) → опросник showed inflated numbers.

**Fix applied:**
```sql
DELETE FROM bot_memory_messages WHERE id IN (4295,4296,4297,4298,4299) AND message_type='image';
UPDATE bot_memory_messages SET tags = jsonb_build_object('building', 'Общежитие') WHERE id IN (4289,4290,4291,4292,4293);
```

**Lesson:** Always ask the user for the real photo count. Never deduplicate "approximately" — partial cleanup (e.g., 10 instead of 5) still shows wrong numbers in the опросник.

## 6. Finding real photo message IDs via Evolution API findMessages

When DB has stale/wrong photo message IDs (from SIM_DATE flood), find the real ones:

**API endpoint:**
```
POST /chat/findMessages/alikhan
Body: {"where": {"key": {"remoteJid": "120363179621030401@g.us"}}, "limit": 50, "page": 1}
Headers: apikey, Content-Type: application/json
```

**Response structure:**
```
data['messages']['records']  — list of messages, paginated
records[i]['key']['id']      — WhatsApp message ID
records[i]['key']['fromMe']  — true if bot, false if user
records[i]['messageType']    — 'imageMessage' for photos
records[i]['messageTimestamp'] — real WhatsApp timestamp
records[i]['message']['imageMessage']['caption'] — building tag
```

**Filtering:** `'imageMessage' in msg.get('message', {})` — check for image messages. Sort by timestamp descending to get most recent. User photos: `fromMe=False`. Paginate with page=1,2,3... to go further back.

**Real case:** DB had 5 stale IDs (ts ~178282xxxx). API found 14 real images; first 5 with Общежитие caption (ts ~178288xxxx) were today's photos.

## 7. main_waha.py: photo deduplication guard

**File:** `main_waha.py:214-221`

**Bug:** No duplicate check before INSERT — seen_id flood re-inserts old photo message IDs into `bot_memory_messages`.

**Fix:** SELECT before INSERT:
```python
cur.execute("SELECT 1 FROM bot_memory_messages WHERE content = %s", (mid,))
if not cur.fetchone():
    cur.execute("""INSERT INTO bot_memory_messages ...""", ...)
else:
    print(f"[PHOTO] Skip duplicate: {mid[:12]}...")
```

**Why not UNIQUE constraint:** `content` column stores long strings (file paths + message IDs). PostgreSQL index row limit is 8191 bytes — some entries exceed this.

## 8. SIM_DATE root cause: all photos get midnight timestamp

**File:** `main_waha.py:217`
```python
datetime.now() if not SIM_DATE else datetime.strptime(SIM_DATE, "%Y-%m-%d")
```

When `SIM_DATE = "2026-06-30"`, ALL image messages get `created_at = 2026-06-30 00:00:00`. Old flood photos and today's real photos share the same timestamp → `fill_ejo.py` can't distinguish them.

**Consequence chain:** seen_ids clear → flood re-processes old messages → old photos re-inserted with SIM_DATE → `fill_ejo.py` photos() queries by date → gets mix of old + new → wrong photos in report.

**Mitigation:** Deduplication guard (section 7) prevents re-insertion. Root fix would separate message storage date from report date but requires broader refactor.
