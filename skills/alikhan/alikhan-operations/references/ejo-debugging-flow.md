# EJO Debugging Flow — отчёт неправильно заполнен

Выучено 15.07.2026. Пользовательская коррекция: агент проверял ЕЖО не в том порядке.

## Правильный порядок

Когда пользователь говорит «отчёт заполнен неправильно»:

```
1. WHATSAPP GROUPS (Evolution API) → какие данные пользователь реально подал
2. DB (bot_memory_facts)             → какие факты сохранил бот
3. EJO (.xlsx)                       → как данные легли в шаблон
```

**Никогда не начинать с ЕЖО.** Сначала понять ЧТО должны были положить, потом ЧТО положили, потом КАК легло.

## Почему этот порядок

- Evolution API показывает полную картину сообщений, включая те что НЕ сохранились в БД
- bot_memory_messages может быть неполным (баг сохранения)
- bot_memory_facts может содержать дубликаты или неправильно распарсенные данные
- ЕЖО — производная от фактов, не источник истины

## Коррекция пользователя (verbatim)

> сначала в песочницу посмотри, ты еще до боевой не дорос. Если я говорю что отчет не правильно заполнен, ты должен смотреть сначала в группу, какие я дал данные, потом в БД, что заполнил бот Алихан, потом в ЕЖО если он сформирован

## Дополнительно: поиск персонала

Персонал в ЕЖО находится в колонках N-P, строки 10-16, НЕ в строках 9-13 (там статистика ТБ).
Искать ПО ВСЕМУ листу, а не в предположительной зоне.

## Пример: баги 15.07.2026

### AL-004 — Планы на завтра в суточных фактах

**Сообщение:** «Работы 3.1.1 - 50 м2 ... Планы на завтра 3.1.1 - 41,8 м2 ...»

**Что пошло не так:** `qa.py` regex `_extract_vor_codes()`: `\s*` между префиксом и VOR-кодом матчит только пробелы. Строка «Планы на завтра  3.1.1» — после «Планы» идёт « на завтра  » (буквы), `\s*` ломается. VOR-код матчится без префикса → category='объём' вместо 'план'.

**Фикс:** `\s*` → `[\w\s]*?` (lazy match word+whitespace). `is_plan` detection: добавить `завтра`.

### AL-005 — Сообщения песочницы не в БД

**Симптом:** Evolution API показывает 6 сообщений в песочнице, bot_memory_messages — только 1.

**Корень:** age gate (5 мин, L517-522 main_waha.py) выполняется ДО `save_message()` (L868). Старые сообщения добавляются в `seen` и пропускаются — никогда не доходят до сохранения.

**Фикс:** `_log_msg()` вызывается сразу после извлечения текста, до всех `continue`-ов (включая age gate).

### AL-011 — Кумулятив потерян (очистка P/S перед чтением)

**Симптом:** Месячный факт = только сегодняшняя работа (50 вместо 900).

**Корень:** clear logic очищает кумулятивные колонки (16=P, 19=S, 21=U) перед чтением. `prev_p = parse_val(ws.cell(r,16).value)` → None → 0. `yesterday_cum()` возвращает (0,0) без вчерашнего файла. Итог: P/S = 0 + сегодня.

**Фикс:** не очищать кумулятивные колонки. Шаблон — источник истины (из corrected EJO).

### AL-012 — Кумулятив двойной счёт (prev_p+v)

**Симптом:** P=950 вместо 900, Q=101% вместо 96%, U=-8.2 вместо 41.8.

**Корень:** Шаблон из corrected EJO уже включает сегодняшнюю работу (P=900). Код добавляет v повторно: `round(prev_p+v, 2)` → 900+50=950.

**Фикс:** писать `round(prev_p, 2)` без добавления v. Кумулятив = шаблон как есть.
