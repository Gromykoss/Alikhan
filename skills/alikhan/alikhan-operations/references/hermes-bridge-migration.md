# Hermes Bridge Migration — July 15, 2026

## Context

Alikhan was migrated from Evolution API (Docker container, REST API on :8080) to Hermes native WhatsApp bridge (Baileys, built-in, REST API on :3000).

## Why

| Before | After |
|--------|-------|
| 3 Docker containers (api, redis, postgres) | 1 container (postgres only) |
| Evolution API maintenance (updates, DB collisions) | Auto with `hermes update` |
| Separate WhatsApp session management | Built into Hermes gateway |
| sendMedia sent to PENDING silently | Bridge sends directly |

## Architecture Change

```
Before: WhatsApp → Evolution API (:8080) → main_waha.py (polling)
After:  WhatsApp → Hermes bridge (:3000) → main_waha.py (polling)
```

The bridge is a Node.js process at `~/.hermes/hermes-agent/scripts/whatsapp-bridge/bridge.js`. It provides:
- GET `/messages` — poll for new messages (empty queue clears on each read)
- POST `/send` — send text message `{chatId, message}`
- POST `/send-media` — send file `{chatId, filePath, mediaType, caption, fileName}`
- GET `/chat/:id` — chat/group info
- GET `/health` — connection status

## Migration Approach: Thin Wrapper (Monkey-Patch)

Instead of rewriting the entire 900-line main_waha.py, we created `bridge_wrapper.py` — a thin translation layer that:

1. Exports `EVO`, `KEY`, `SANDBOX`, `PRODUCTION` globals (so existing code doesn't break)
2. Monkey-patches `requests.post` to intercept Evolution API calls and redirect to bridge
3. Monkey-patches `urllib.request.Request/urlopen` for media endpoints
4. Translates bridge message format to Evolution API format

This approach preserves all business logic (QA, EJO, polls, calendar) untouched.

### Wrapper Pattern

```python
# bridge_wrapper.py — import at top of main_waha.py
from bridge_wrapper import *
```

The wrapper intercepts:
- `requests.post(url)` when URL contains `/chat/findMessages/` → bridge `GET /messages`
- `requests.post(url)` when URL contains `/message/sendText/` → bridge `POST /send`
- `urllib.request.Request(url)` when URL contains `/message/sendMedia/` → bridge `POST /send-media`
- `urllib.request.Request(url)` when URL contains `/chat/getBase64FromMediaMessage/` → no-op

### Key Message Format Translation

Bridge format:
```json
[{"chatId": "X@g.us", "senderId": "79...@s.whatsapp.net", "body": "text", "timestamp": 1784..., "isGroup": true}]
```

Translated to Evolution API format:
```json
{"messages": {"records": [{"key": {"id": "1784..."}, "message": {"conversation": "text"}, "messageTimestamp": 1784...}]}}
```

## Pairing WhatsApp

### Prerequisites

1. Stop Evolution API: `docker stop evolution-api`
2. Need to patch bridge.js for group message allowlist (one line):

```js
// bridge.js line 637: allow ALL group senders, not just whitelisted
if (!isGroup && WHATSAPP_DM_POLICY !== 'pairing' && !matchesAllowedUser(...)) {
```

### Starting the Bridge

```bash
cd ~/.hermes/hermes-agent/scripts/whatsapp-bridge
WHATSAPP_MODE=bot node bridge.js --session ~/.hermes/sessions/whatsapp
```

This shows an ASCII QR code in terminal. Scan with WhatsApp → Linked Devices.

### QR Code Delivery (PITFALL)

**Problem:** The ASCII QR generated by `qrcode-terminal` in the bridge can't be delivered through Telegram as an image. `image_generate` can't create scannable QR codes from local files.

**Solutions tried (both failed):**
- `MEDIA:/path/qr.png` — doesn't deliver on Telegram
- `image_generate(image_url=...)` — FAL.ai can't access VPS local files
- HTTP server on port 8080 — port not exposed to internet after Docker removed

**Working approach:** User must run the bridge command directly in Hostinger Web Terminal and scan the ASCII QR from screen.

### Post-Pairing Verification

```bash
curl http://127.0.0.1:3000/health
# Should return: {"status":"connected",...}

curl http://127.0.0.1:3000/chat/120363179621030401@g.us
# Should return group metadata
```

## Pitfalls Noted During Migration

1. **Bridge only starts when WhatsApp session connects.** After `hermes gateway restart`, bridge won't auto-start if no WhatsApp session exists.
2. **One WhatsApp number = one active session.** Unlinking Evolution API kicks it off — migration must be done sequentially.
3. **Bridge `/messages` endpoint drains queue on each read.** Multiple listeners (sandbox + production) must share state carefully.
4. **Evolution API PostgreSQL must stay running** — bot still uses `bot_memory_messages` and `bot_memory_facts` tables.
5. **Production listener sees empty bridge queue** — bridge starts fresh with no history, so old production messages are invisible.
