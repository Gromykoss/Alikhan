# Alikhan EJO Fill Rules

## Источники данных (OJR — 2026-07-18)

**Миграция на OJR-таблицы завершена.** Все функции чтения данных в `fill_ejo.py` теперь используют OJR как primary source с graceful fallback на `bot_memory_facts`.

| Функция | Primary source | Fallback | Также пишет |
|---------|---------------|----------|------------|
| `weather()` | Open-Meteo API | — | `ojr_weather` (через `save_weather()`) |
| `incidents()` | `ojr_incidents` via `get_daily_incidents()` | `bot_memory_facts` | — |
| `volumes()` | `ojr_section3_work_log` via `get_daily_works()` | `bot_memory_facts` regex | — |
| `staff()` | `ojr_section1_personnel` via `get_daily_personnel()` | `bot_memory_facts` | — |
| `photos()` | `ojr_photo_log` → `bot_memory_messages` | Оба источника проверяются | — |

**Graceful fallback:** каждая OJR-читающая функция обёрнута в `try/except` с логом `[FUNC OJR ERR] ... falling back to legacy`. При отсутствии OJR-таблиц код не ломается. Детали роутинга: `skill_view('alikhan-operations', file_path='references/ojr-migration-routing.md')`.

## Колонки листа «Ежедневный отчет»

| Кол | Excel | Назначение | Формула | Формат |
|-----|-------|------------|---------|--------|
| L | 12 | План на сутки | `= v` | число |
| M | 13 | Факт за сутки | `= v` | число |
| N | 14 | % за сутки | `= 1` (всегда 100% — L=M) | `0%` |
| O | 15 | План на месяц | из шаблона | число |
| P | 16 | Накоп. с начала месяца | `= prev_P + v` | число |
| Q | 17 | % от мес. плана | `= P / O` | `0%` |
| R | 18 | Общий план | из шаблона | число |
| S | 19 | Накоп. с начала проекта | `= prev_S + v` | число |
| T | 20 | % от общего плана | `= S / R` | `0%` |
| U | 21 | Остаток | `= O − P` | число |

## Логика скрытия строк (только в сгенерированном ЕЖО)

| Условие | Видна |
|---------|-------|
| Есть работа сегодня (L или M) | ✅ |
| O > 0 И U > 0 (план + остаток) | ✅ |
| Заголовки секций/подсекций (2, 3.1...) | ✅ |
| Фаза 8 (нет работ, нет плана) | ❌ скрыта |
| Остальное без работ и без остатка | ❌ скрыто |

## Лист «Персонал и техника»

- **АйБиКон** (строки 8-13): из табеля через локальный кеш (`/tmp/hermes-media-cache/`). Руководитель всегда 1.
- **Субподрядчики** (строки 14-32): из QA-фактов категории «персонал»
- Табель: читается из `local_path` в тегах БД, при дубликате обновляется путь

## Лист «Материалы и планы»

- **Планы на завтра**: заголовки зданий на строках 14, 17, 22. Пункты ниже.
- Парсинг планов: из сырых сообщений БД (fallback, если Grok не выделил «план» как категорию)
- «План» определяется по позиции ДО кода (не после — `3.1.5 = 142Планы` ≠ план)

## Отправка ЕЖО

- Файл > 10MB → `_send_document()` через bridge `requests.post`
- Bridge: `express.json({limit: '50mb'})` в `bridge.js`

## Питфоллы

- ❌ N ≠ 1 → L всегда = M, поэтому N всегда 100%
- ❌ `prev_p` без `+ v` → кумулятив не растёт
- ❌ Скрытие строк с U>0 → работа запланирована, должна быть видна
- ❌ Дубликаты записи колонок → каждая пишется ОДИН раз
- ❌ `0.0"%"` → использовать стандартный `0%`
- ❌ Парсинг «план» по факту целиком → проверять позицию ДО кода
- ❌ Бот шлёт старый кеш → удалить `/tmp/ЕЖО_{date}_v*.xlsx` перед перегенерацией
- ❌ Планы из QA-фактов → если Grok не выделил категорию «план», парсить сырые сообщения БД (`ILIKE '%%план%%'`)
- ❌ Bridge migration: photo handler ищет `imageMessage` (Evolution) вместо `_media` (Bridge). Всегда проверять wrapper перед патчем main_waha.
