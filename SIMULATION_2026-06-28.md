# ЕЖО Pipeline — симуляция 28.06.2026

## Дата
2026-06-28 (симуляция через SIM_DATE)

## Архитектура цикла ЕЖО

```
QA-опрос (вечер) → QA-факты в PostgreSQL
                         ↓
          fill_ejo.py (авто-генерация)
                         ↓
          /tmp/ЕЖО_2026-06-28_vN.xlsx
                         ↓
          Ручная правка пользователем
                         ↓
          Отправка в песочницу WhatsApp
                         ↓
          main_waha.py: сравнение с авто, сохранение шаблона
                         ↓
          templates/ЕЖО_шаблон.xlsx + /tmp/ЕЖО_2026-06-28_v1.xlsx
                         ↓
          Следующий день: генерация из обновлённого шаблона
```

## Методика симуляции

1. **Загрузка исторических данных:** QA-факты за 27.06 и 28.06 в PostgreSQL
2. **Авто-генерация 28.06:** `python3 fill_ejo.py 2026-06-28`
3. **Ручная правка:** добавление кодов 4-го уровня (раскрытие подкодов), исправление накопленных
4. **Отправка правленого файла:** бот принимает, сравнивает с авто, обновляет шаблон
5. **Повторная генерация:** из обновлённого шаблона — проверка, что накопленные не удваиваются

## Правки кода

### 1. Персонал по профессиям из табеля (fill_ejo.py)
- **Было:** `get_aibikon_headcount()` возвращал `int` (голое число)
- **Стало:** `{'total': N, 'by_prof': {'профессия': кол-во, ...}}`
- **Источник:** колонка C (Должность) табеля, фильтр по дате (колонка 5+день−1, значение=8)
- **Mapping профессий:** `PROF_MAP` — рук.проекта→Руководителя строительства, геодезист→Инженер геодезист, тб→Инженер ТБ и ОТ, пто→Инженер ПТО, электрик→Электрик
- **Заполнение:** лист «Персонал и техника», строки 9-13 (вместо хардкода)
- **Лист «Ежедневный отчет»:** M11/M12/M17/M18 используют `aibikon['total']`

### 2. Логотип — удаление дублирования (fill_ejo.py)
- **Было:** логотип добавлялся в двух местах: в начале `fill()` и после `save()`
- **Стало:** удалены оба добавления (логотип уже в шаблоне)

### 3. Накопленные значения — защита от удвоения (fill_ejo.py)
- **Было:** `prev_p = max(template_month_fact, yesterday_cum)` — при повторной генерации шаблон уже содержит сегодняшние данные → удвоение
- **Стало:** если `v > 0` (есть сегодняшние работы), `prev = yesterday_cum()` (вчерашний файл). Шаблон используется только для кодов без сегодняшних работ
- **data_only=True:** загрузка шаблона с кешированными значениями (формулы → значения) — нетронутые ячейки не теряются при save

### 4. Сравнение шаблона по кодам, а не по строкам (main_waha.py)
- **Было:** `_update_template_from_correction()` сравнивал строка-в-строку (r=24, 25, 26...). Добавление строки → всё съезжает → сотни ложных отличий
- **Стало:** построение code→values map для обоих файлов, сравнение по коду ВОР (колонка C). Вставка/удаление строк не ломает diff
- **Результат:** с 672 отличий до 49 (реальных), затем до 0

### 5. Дата из имени файла (main_waha.py)
- **Было:** `date_str = datetime.now()` — все правленые файлы сохранялись с сегодняшней датой
- **Стало:** regex `(\d{2})\.(\d{2})\.(\d{4})` из имени файла (например, «ЕЖО_27.06.2026 АйБиКон.xlsx» → «2026-06-27»)
- **Сохранение:** `/tmp/ЕЖО_{date}_v1.xlsx` (всегда перезаписывается) + `/tmp/ЕЖО_template_{date}.xlsx`

### 6. Сортировка авто-файлов по mtime (main_waha.py)
- **Было:** `sorted(files, reverse=True)` — строковая сортировка, «29» > «28»
- **Стало:** `sorted(files, key=os.path.getmtime, reverse=True)` — последний сгенерированный файл

## Ключевые принципы

1. **Шаблон — источник истины:** после пользовательской правки шаблон обновляется и служит основой для следующего дня
2. **Вчерашний файл — источник prev:** `yesterday_cum()` ищет `/tmp/ЕЖО_{yesterday}_v1.xlsx` (сохранённый после правки)
3. **Сравнение по кодам:** вставка строк не ломает diff
4. **Формулы → значения:** `data_only=True` при загрузке шаблона предотвращает потерю данных
5. **Дата из имени файла:** бот автоматически определяет дату отчёта
