## 1. Формат SKILL.md

### 1.1 Структура файла

```
категория/
  название-навыка/
    SKILL.md              ← основной файл навыка
    references/           ← справочные материалы
      fix-2026-07-01.md
      schema.sql
      ...
    scripts/              ← исполняемые скрипты
      health_check.py
      ...
    templates/            ← шаблоны (если нужны)
```

### 1.2 Frontmatter (YAML)

Минимальный обязательный набор:

```yaml
---
name: alikhan-fill-ejo
description: Полная логика заполнения и формирования ЕЖО — все колонки, скрытие строк, персонал, планы, отправка.
---
```

Расширенный (рекомендуемый) набор:

```yaml
---
name: alikhan-operations
description: Operational patterns for the Alikhan WhatsApp bot
metadata:
  tags: [alikhan, whatsapp, db, schedule, operations]
  category: operations
  phase: maintenance
  quality_tier: evidence-gated
triggers:
  - User asks about Alikhan, WhatsApp bot, алихан
  - Fixing/deploying WhatsApp AI assistant
version: 2.19.0
---
```

| Поле | Назначение | Обязательно |
|------|-----------|-------------|
| `name` | Уникальное имя навыка (lowercase + hyphens) | ✅ Да |
| `description` | Одна строка — что делает навык | ✅ Да |
| `version` | Семантическая версия (MAJOR.MINOR.PATCH) | Рекомендовано |
| `triggers` | Когда агент должен загрузить навык | Рекомендовано |
| `metadata.tags` | Теги для поиска | Опционально |
| `metadata.category` | Категория навыка | Опционально |
| `metadata.quality_tier` | Уровень качества: `evidence-gated`, `verified`, `draft` | Опционально |

### 1.3 Структура тела (markdown)

Рекомендованная структура от общего к частному:

```
# Заголовок (H1) — что делает навык

## Принцип / Golden Rule — главное правило, нарушение которого = провал

## Архитектура — общая схема, поток данных, ключевые файлы

## ИСТОЧНИК ДАННЫХ / Data Sources — таблицы БД, API, файлы

## Workflow / Flow — пошаговый процесс

## Команды / Commands — таблица триггеров и действий

## Ключевые файлы — пути к файлам проекта

## Pitfalls — пронумерованный список ошибок

## Проверка / Verification — runnable shell-команды

## Связанные навыки — перекрёстные ссылки
```

**Пример из alikhan-daily-snapshot:**
- Принцип: разделение движков (xAI для vision, Ollama для текста)
- Архитектура: таблица источников данных с лимитами
- Pitfalls: 10+ пунктов с симптомами
- Проверка: `tail -f ... | grep SNAPSHOT`

## 2. Как писать trigger conditions

### 2.1 Три типа триггеров

**Тип 1 — явная команда пользователя:**
```yaml
triggers:
  - «начать опрос» / «закончить опрос» / «сформируй ЕЖО»
  - «снимок дня», «итоги дня», «сводка дня», «дайджест»
```

**Тип 2 — тематический контекст:**
```yaml
triggers:
  - User asks about Alikhan, WhatsApp bot, алихан
  - Fixing/deploying WhatsApp AI assistant
  - заполнение ежедневного отчёта по стройке
```

**Тип 3 — автоматический (по событию):**
```yaml
triggers:
  - Любое фото в WhatsApp-группе (песочница или прод).
```

### 2.2 Правила хороших триггеров

1. **Конкретность важнее полноты.** Лучше 3 точных триггера чем 15 размытых.
2. **Русский + английский.** Пользователи пишут на обоих языках.
3. **Вариации словоформ.** «опрос» / «опроса» / «опросом» — используй корни: `опрос`.
4. **Синонимы.** «запусти опрос» = «начать опрос» = «запускай опрос» — перечисли все.

**Плохой пример:**
```yaml
triggers:
  - stuff about EJO
```
**Хороший пример:**
```yaml
triggers:
  - «Алихан раскрой отчет» / «Алихан запускай опрос» / «Алихан формируй отчет»
  - ЕЖО, ежедневный отчёт, заполнение шаблона
  - Poll lifecycle: start, close, status, residuals
```

## 3. Как писать пошаговые инструкции

### 3.1 Формат: нумерованный список + код

Каждый шаг = одно действие + команда для его выполнения:

```markdown
## Месячный цикл

### Начало месяца — план
1. «Алихан раскрой отчет» → бот присылает шаблон со всеми строками
2. Пользователь заполняет O (план) и U (остаток) → присылает файл обратно
3. Бот сохраняет шаблон. **Опрос НЕ закрывается.**

### Ежедневный опрос
1. «Алихан запускай опрос» → бот: **O > 0 ∧ U > 0**
2. QA-сводка + список работ с остатками
3. Прорабы: `код = объём` (только где были работы сегодня)
4. «Алихан закончить опрос» → ЕЖО
```

### 3.2 Таблицы для команд и источников

```markdown
| Команда | Действие |
|---------|----------|
| `Алихан раскрой отчет` | Шаблон со всеми строками |
| `Алихан запускай опрос` | Опрос O>0 ∧ U>0 |
| `Алихан статус опроса` | Сводка |
| `Алихан формируй отчет` | Закрыть опрос → ЕЖО |
```

### 3.3 Ключевые принципы

1. **Один шаг = одно действие.** Не «проверь БД и потом сгенерируй ЕЖО» — раздели.
2. **Код всегда в блоках.** Не inline.
3. **Команды — в backticks.** `grep`, `SELECT`, `systemctl`.
4. **Результат шага виден.** Указывай что должно произойти после выполнения.
5. **Условные ветвления — явно.** «Если poll активен → ... Иначе → ...»

## 4. Как добавлять pitfalls (типовые ошибки)

### 4.1 Структура питфолла

Каждый питфолл содержит **обязательные 3 элемента**:

```markdown
### N. 🐉 Краткое название (дата)

**Symptom:** Что видит пользователь / что идёт не так

**Root cause:** Почему это происходит (техническая причина)

**Fix:** Как исправить (код, команда, SQL)
```

**Пример из alikhan-maintenance:**

```markdown
### 12. Photo vision silently broken — import `ask_ollama_raw` doesn't exist (FIXED 2026-07-18)

**Symptom:** Фото приходят, сохраняются в БД, но vision-описание не генерируется.
Лог: `[PROD PHOTO DESC ERR]`

**Root cause:** `main_waha.py` импортировал `from handlers import ask_ollama_raw`,
но в `handlers.py` существует только `ask_grok_raw`.
Импорт внутри `try/except` → `NameError` молча глотается.

**Fix:** `from handlers import ask_ollama_raw` → `from handlers import ask_grok_raw`
в ОБОИХ местах (sandbox + production).
```

### 4.2 Приоритеты питфоллов

| Приоритет | Маркер | Критерий |
|-----------|--------|----------|
| P0 — Критический | 🐉 | Данные теряются, бот падает, пользователь не получает результат |
| P1 — Высокий | ⚠️ | Функциональность нарушена, но есть обходной путь |
| P2 — Средний | 📝 | Некорректное отображение, не влияет на данные |
| P3 — Низкий | 💡 | Улучшение, оптимизация |

### 4.3 Правила хороших питфоллов

1. **Дата в названии.** «P0. Три этапа — не мешать (16.07.2026)» — сразу видно когда обнаружено.
2. **Реальные примеры.** Не «может возникнуть X», а «14.07.2026 агент изменил формулу 26%→94%».
3. **Диагностика перед фиксом.** Сначала как найти проблему, потом как исправить.
4. **Проверка после фикса.** `grep`, `tail`, `SELECT` — команда которая подтверждает исправление.
5. **Нумерация сохраняется.** P0, P1, P2... даже если промежуточные удалены.

### 4.4 Антипаттерны в pitfalls

❌ **Размытый:**
```
- Иногда бот не отвечает
```
✅ **Конкретный:**
```
### 8. `int(None)` crash in poll.py format strings (FIXED 2026-07-07)
**Symptom:** `build_poll_message()` вызывает `TypeError: int() argument must be...`
**Root cause:** Три места в poll.py вызывали `int()` на None из БД
**Fix:** Добавлены `is not None` guards перед каждым `int()`
```

## 5. Как тестировать навык перед публикацией

### 5.1 Чеклист тестирования

- [ ] **Frontmatter валиден:** `name`, `description` заполнены, YAML без ошибок
- [ ] **Все команды работают:** каждая команда из раздела «Commands» протестирована в песочнице
- [ ] **Питфоллы проверены:** каждый питфолл содержит symptom + root cause + fix
- [ ] **Верификация воспроизводима:** команды из раздела «Verification» запускаются и дают ожидаемый результат
- [ ] **Пути актуальны:** все пути к файлам (`/home/hermes-workspace/...`) существуют
- [ ] **Связанные навыки указаны:** перекрёстные ссылки на другие навыки корректны
- [ ] **Нет мёртвых ссылок:** все `references/*.md` существуют в директории навыка

### 5.2 Тестовый прогон

```bash
# 1. Валидация YAML
python3 -c "import yaml; yaml.safe_load(open('SKILL.md').read().split('---')[1])"

# 2. Проверка путей
grep -oP '(?<=`)[^`]+\.(py|md|sql|sh|xlsx)(?=`|$)' SKILL.md | while read f; do
  [ -f "$f" ] || echo "MISSING: $f"
done

# 3. Проверка references
for ref in references/*.md; do
  grep -q "$(basename $ref)" SKILL.md || echo "UNREFERENCED: $ref"
done
```

### 5.3 Критерии приёмки

Навык готов к публикации когда:
1. Агент, загрузивший навык, выполняет задачу **с первого раза без ошибок**
2. Все pitfalls покрывают **реальные провалы**, а не гипотетические
3. Verification-команды дают **воспроизводимый результат** на чистом окружении

## 6. Как обновлять навык после изменений в коде

### 6.1 Правило: навык следует за кодом

Изменение в коде → немедленное обновление навыка. Не «потом», не «на следующей неделе».

### 6.2 Процесс обновления

1. **Зафиксировать изменение в CHRONOLOGY.md проекта**
2. **Обновить SKILL.md:**
   - Изменившиеся пути к файлам
   - Новые/изменённые команды
   - Новые pitfalls (баги, обнаруженные при изменении)
   - Обновить `version` во frontmatter
3. **Добавить reference если изменение крупное:**
   ```bash
   # Создать references/fix-2026-07-18.md с описанием изменения
   ```
4. **Проверить перекрёстные ссылки** — другие навыки могут ссылаться на изменившуюся логику

### 6.3 Инкремент версии

| Изменение | Инкремент |
|-----------|-----------|
| Исправление опечатки, уточнение формулировки | PATCH (1.0.0 → 1.0.1) |
| Новый pitfall, новая команда, новый reference | MINOR (1.0.0 → 1.1.0) |
| Новая архитектура, миграция БД, смена API | MAJOR (1.0.0 → 2.0.0) |

### 6.4 Пример: миграция Evolution API → Hermes Bridge

```markdown
## Обновление навыка при миграции (15.07.2026)

### Что изменилось:
- Evolution API (:8080) → Hermes Bridge (:3000)
- `bridge_wrapper.py` monkey-patches requests/urllib
- Документы скачиваются через `/tmp/hermes-media-cache/`

### Что обновить в SKILL.md:
1. Пути: `EVO` → `BRIDGE`, порт 8080 → 3000
2. Команды: `evolution-api` → `hermes-whatsapp-bridge`
3. Verification: `curl :8080` → `curl :3000/health`
4. Pitfalls: добавить «Bridge crash: Timeout in AwaitingInitialSync»
5. References: добавить `hermes-bridge-migration.md`
```

## 8. Чеклист создания нового навыка

### Перед написанием

- [ ] Задача решена минимум 2 раза (навык = паттерн, не одноразовый скрипт)
- [ ] Определён scope: что ВХОДИТ в навык, что НЕ входит
- [ ] Определена категория (alikhan / project / projects / devops)
- [ ] Собраны все пути, команды, имена таблиц

### При написании

- [ ] Frontmatter: name, description, triggers, version
- [ ] H1 заголовок — что делает навык
- [ ] Принцип / Golden Rule — одно предложение
- [ ] Архитектура — схема или таблица
- [ ] Workflow — нумерованные шаги
- [ ] Команды — таблица триггер → действие
- [ ] Pitfalls — минимум 3 реальных проблемы
- [ ] Verification — воспроизводимые shell-команды
- [ ] Связанные навыки — перекрёстные ссылки

### После написания

- [ ] Все пути проверены (`grep` + `test -f`)
- [ ] Все references существуют в `references/`
- [ ] Команды Verification запущены и дали ожидаемый результат
- [ ] Навык загружен через `skill_view()` и агент выполнил задачу
- [ ] CHRONOLOGY.md обновлён

## 9. Антипаттерны (как НЕ надо)

| Антипаттерн | Почему плохо | Как исправить |
|-------------|-------------|---------------|
| Навык без pitfalls | Агент повторит те же ошибки | Добавить минимум 3 реальных провала |
| «Сделай ЕЖО» без шагов | Агент пропустит проверку данных | Разбить на: проверь БД → проверь QA → заполни → проверь → отправь |
| Пути без проверки | Навык ссылается на несуществующие файлы | `grep` + `test -f` для каждого пути |
| Pitfalls без дат | Непонятно актуальна ли проблема | Добавить дату в название: `(FIXED 2026-07-18)` |
| Verification без команд | Агент не знает как проверить | Только runnable shell: `grep`, `curl`, `tail` |
| Навык-простыня без структуры | Агент теряется в тексте | H2/H3 заголовки, таблицы, блоки кода |
| Дублирование между навыками | Расхождение при обновлении | Перекрёстные ссылки вместо копипасты |

## 10. Инструменты

### Создание навыка
```bash
# Через skill_manage (рекомендуется)
skill_manage(action='create', name='my-new-skill', category='alikhan', content='...')

# Или вручную
mkdir -p ~/.hermes/skills/alikhan/my-new-skill/references
```

### Просмотр навыка
```python
skill_view('alikhan-poll-logic')                        # основной SKILL.md
skill_view('alikhan-operations', file_path='references/ojr-schema.md')  # reference
```

### Обновление навыка
```python
skill_manage(action='patch', name='alikhan-poll-logic',
  old_string='старая формула',
  new_string='новая формула')
```

---

> **Вывод:** Хороший навык — это не документация, а **инструкция для агента**, которая предотвращает повторение уже совершённых ошибок. Каждый раздел должен отвечать на вопрос «что делать» и «почему нельзя иначе».
